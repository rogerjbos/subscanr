Substrate blockchain analysis in R

Roger J. Bos

2022-07-19

This vignettes demonstrates the functionality of the subscanr package that facilitates substrate blockchain analysis in R. By substrate we are referring to the Polkadot / Kusama (DotSama) ecosystem, which could include up to 200 different parachains.

The package provides an interface into three different indexing services in the DotSama ecosystem, along with a bonus section showing how to access on-chain data using Python code in R:

  • Subscan
  • Sub Query
  • Polkaholic
  • Python Substrate Interface

Installation

You can install the subscanr package using devtools (or remotes) as follows:

library(devtools)
install_github("rogerjbos/subscanr")
#> Skipping install of 'subscanr' from a github remote, the SHA1 (de9a22ed) has not changed since last install.
#>   Use `force = TRUE` to force installation

Once installed, you can load the package. The package comes with a number of functions which will be detailed below, and it also comes with a few objects and helper functions. I won’t cover all of them, but the tokens object is useful because it shows the number of decimal places used for each token. Subscan

Subscan provides an API for accessing their data. You have to request a key here. I will show some examples of pulling data from the Subscan API below, but if you have any questions, the best place to ask them is in their maxtrix site.

Subscan currently covers different chains, which are stored in the endpoints variable. You can use the get_endpoint() function to get the host address:

head(endpoints)
#>                                      
#> 1 function (x, on = "months", k = 1) 
#> 2 {                                  
#> 3     if (timeBased(x)) {            
#> 4         NR <- length(x)            
#> 5         x <- xts(, order.by = x)   
#> 6     }
get_endpoint("Acala")
#> [1] "acala.api.subscan.io"

The primary function is get_subscan_events, which we will call using the extract = FALSE parameter. The functions returns two objects. The core_data object lists the general details of each requested event, such as the block number, extrinsic_hash, and timestamp. The params object contains the actual details on the event, but it is in JSON format so it is more difficult to read.

tmp <- get_subscan_events(nobs = 10, network = 'Karura', module = 'dex', call = 'Swap', extract = FALSE)

The get_subscan_events function can be called for any endpoint, but the format of the params object will be different for every module and chain. My works involves the Acala and Karura chains, so I have coded an extract function to re-format the JSON output and add it to the core_data object, but this have only been implemented for Acala and Karura. The default value of the extract parameter is TRUE, so if you call the function again without that parameter, you will see the new core_data object.

tmp <- get_subscan_events(nobs = 10, network = 'Karura', module = 'dex', call = 'Swap')
#> [1] "10 rows for page 1/1 Success at 2022-07-19 20:43:05"

# First few lines of the core_data
head(tmp$core_data)
#> NULL

The Subscan API limits each call to 100 rows of data, so if you set nobs to greater than 100 the function will use pagination to make multiple calls and aggregate the data. If you leave module and/or call empty it will return all events. That will give you an idea of what options are available for a specific chain so you can specify them and narrow your search.

tmp <- get_subscan_events(nobs = 300, network = 'Karura', module = '', call = '', extract = FALSE)
#> [1] "100 rows for page 1/3 Success at 2022-07-19 20:43:06"
#> [1] "100 rows for page 2/3 Success at 2022-07-19 20:43:08"
#> [1] "100 rows for page 3/3 Success at 2022-07-19 20:43:09"

# Module and event types returned for the specified chain
unique(tmp$core_data$module_id)

You can use the get_subscan_token function to get a list of all the tokens available on a specific chain.

get_subscan_token(network = 'Acala')
get_subscan_metadata(network = 'Acala')
